# SharedClipboard 产品策略

本文档只描述基于当前代码的产品方向，不再包含外部市场估算或未经验证的竞品结论。

## 1. 当前产品定位

SharedClipboard 是一个本地优先的跨平台剪贴板历史管理器，核心目标是：
- 快速记录和检索文本、图片剪贴板内容。
- 支持多设备同步，但不强迫用户依赖云端。
- 通过插件机制承载文本处理、AI 工具和后续扩展能力。
- 保持系统托盘常驻和边缘停靠体验，尽量低打扰。

## 2. 当前已经落地的能力

从代码看，当前产品已经具备这些能力：
- 剪贴板历史、搜索、收藏、分页和置顶。
- SQLite 本地库，支持 WAL 和 `busy_timeout`。
- MySQL 后端与云端同步入口。
- 设置页中的数据库、过滤、语言、热键和插件管理。
- 插件系统，包括加载、配置、启用/禁用、卸载、商店安装和执行反馈。
- 内置插件 `smart_text` 与 `ai_image_gen`。

## 3. 核心用户假设

这里的用户画像是产品方向上的内部假设，不是市场统计结论：
- 开发者：需要处理文本、代码片段和大量复制粘贴。
- 办公用户：需要快速回溯历史剪贴板、收藏重要内容。
- 设计或内容创作者：需要在文本和图片之间切换。
- 小团队用户：需要本地优先或自建同步方案。

## 4. 差异化方向

代码已经体现出的差异化方向有三条：

### 4.1 本地优先

应用默认能独立运行，数据先落本地库，再根据配置选择 MySQL 或云端同步。这样可以降低启动门槛，也方便个人用户直接使用。

### 4.2 可扩展

插件系统已经不是概念设计，而是实际实现。后续新增能力应优先考虑：
- 文本处理
- 图像处理
- AI 辅助
- 企业定制扩展

### 4.3 可控同步

同步能力已经拆成三个层次：
- `local`
- `mysql`
- `cloud`

这意味着产品策略上可以继续强调“可选同步”和“用户控制”，而不是把云端作为唯一入口。

## 5. 近期优先级

以下是和当前代码最贴近、也最值得继续推进的事项：

1. 补齐启动链路、插件执行和同步链路的集成测试。
2. 继续收紧插件权限说明，特别是云端客户端方法级权限。
3. 优化 `REPLACE`、`SAVE` 等插件结果在 UI 和仓库层的反馈一致性。
4. 改善 Linux 和 Wayland 环境下的兼容体验。
5. 提升错误提示和日志可读性，降低插件安装、配置和同步失败的排障成本。

## 6. 中期方向

如果现有基础继续稳定，后续更适合扩展的方向是：
- 更强的内容组织能力，例如标签、分类和收藏视图。
- 更丰富的插件配置控件。
- 更完整的云端插件商店和版本管理。
- 更细粒度的同步策略和冲突处理。
- 更清晰的企业场景能力，但应建立在现有本地优先架构上。

## 7. 目前不建议写进宣传文案的内容

- 任何未经验证的市场规模数字。
- 任何“已经完成但实际上只是规划”的功能。
- 任何把外部服务当成必需项的叙述。
- 任何和当前实现不一致的插件或产品名称。

## 8. 结论

SharedClipboard 当前最稳妥的产品叙事是：
- 一个能独立运行的跨平台剪贴板历史工具。
- 具备本地优先的数据存储和可选同步。
- 通过插件系统持续扩展能力。

接下来的产品策略应该围绕“稳定、可扩展、可控同步”继续推进，而不是先写大而全的市场故事。
